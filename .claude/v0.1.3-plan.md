## 1. GitHub Sync Adapter

### The Problem

Current sync adapters (Notion, Google Docs) sync **individual notes** to individual remote documents. GitHub is fundamentally different: a **directory of notes** maps to a **repository of files**, and changes should produce **git commits**, not silent overwrites. The existing `SyncAdapter` trait operates at note-level granularity, which doesn't fit the repo-as-a-unit model.

### Key Insight: Directory-Level Sync Config

Instead of attaching a `SyncConfig` to each note, we attach a **single config to a vault directory**. The adapter then manages the entire subtree:

```
vault/projects/prism/    ←→    github.com/omniharmonic/prism-docs
  README.md                      README.md
  architecture.md                architecture.md
  notes/design.md                notes/design.md
```

### Architecture

#### 1a. New Trait: `DirectorySyncAdapter`

The existing `SyncAdapter` trait is note-level. GitHub needs a **directory-level** companion:

```rust
// src-tauri/src/sync/adapters/mod.rs

#[async_trait]
pub trait DirectorySyncAdapter: Send + Sync {
    /// Sync an entire directory of notes to/from a remote repo.
    /// Returns per-file results.
    async fn sync_directory(
        &self,
        notes: &[Note],
        config: &DirectorySyncConfig,
        parachute: &ParachuteClient,
    ) -> Result<DirectorySyncResult, PrismError>;

    /// Initialize remote (clone or create repo)
    async fn init_remote(
        &self,
        config: &DirectorySyncConfig,
    ) -> Result<(), PrismError>;

    /// Check remote status (ahead/behind/diverged)
    async fn remote_status(
        &self,
        config: &DirectorySyncConfig,
    ) -> Result<RemoteStatus, PrismError>;
}
```

#### 1b. Data Model

```rust
// src-tauri/src/models/sync_config.rs (extend)

pub struct DirectorySyncConfig {
    pub id: String,                    // UUID
    pub vault_path: String,            // e.g. "vault/projects/prism-docs"
    pub adapter: String,               // "github"
    pub remote_url: String,            // "https://github.com/omniharmonic/prism-docs"
    pub branch: String,                // "main"
    pub local_clone_path: PathBuf,     // ~/.config/prism/sync/github/{repo-name}
    pub auth: GitAuthConfig,           // token or SSH key path
    pub commit_strategy: CommitStrategy,
    pub conflict_strategy: String,     // "local-wins" | "remote-wins" | "ask"
    pub last_synced: String,           // ISO timestamp
    pub auto_sync: bool,              // commit+push on save?
    pub file_extension: String,        // default ".md" — appended to notes without extensions
    pub ignore_patterns: Vec<String>,  // glob patterns to skip (e.g. "*.tmp")
}

pub enum CommitStrategy {
    /// One commit per save (real-time sync feel)
    PerSave,
    /// Batch all changes since last sync into one commit
    Batched,
    /// Manual — user triggers sync explicitly
    Manual,
}

pub enum GitAuthConfig {
    Token(String),         // GitHub PAT
    SshKey(PathBuf),       // Path to SSH private key
}

pub struct DirectorySyncResult {
    pub pushed: Vec<String>,    // note IDs that were pushed
    pub pulled: Vec<String>,    // note IDs that were pulled/created locally
    pub conflicts: Vec<FileSyncConflict>,
    pub errors: Vec<(String, String)>,  // (path, error message)
}

pub struct FileSyncConflict {
    pub path: String,
    pub local_content: String,
    pub remote_content: String,
}

pub enum RemoteStatus {
    UpToDate,
    LocalAhead(u32),    // N commits ahead
    RemotAhead(u32),    // N commits behind
    Diverged { ahead: u32, behind: u32 },
}
```

#### 1c. GitHub Adapter Implementation

**Rust crate:** Use `git2` (libgit2 bindings) for all git operations. Add to Cargo.toml:
```toml
git2 = "0.19"
```

**File:** `src-tauri/src/sync/adapters/github.rs`

**Core flow — `sync_directory()`:**

```
1. Ensure local clone exists (clone if not)
2. git fetch origin
3. Compare local branch vs remote:
   a. For each note in the vault directory:
      - Map vault path → repo file path (strip vault prefix, ensure .md extension)
      - Compare note content vs file on disk
      - If note is newer → stage file change
      - If file on disk is newer (from remote) → update note in Parachute
   b. Detect deleted notes → delete files (or vice versa)
4. If any staged changes:
   - git commit with descriptive message
   - git push origin {branch}
5. Return DirectorySyncResult
```

**Path mapping logic:**
```
Vault path: "vault/projects/prism-docs/architecture.md"
Sync config vault_path: "vault/projects/prism-docs"
→ Strip prefix: "architecture.md"
→ Repo file: "{clone_path}/architecture.md"

Vault path: "vault/projects/prism-docs/notes/design"  (no extension)
→ Strip prefix: "notes/design"
→ Repo file: "{clone_path}/notes/design.md"  (add configured extension)
```

**Commit message generation:**
- `PerSave`: `"Update {filename}"` or `"Add {filename}"` or `"Delete {filename}"`
- `Batched`: `"Sync {N} files from Prism vault"` with file list in body
- Include Prism metadata in commit trailer: `Prism-Sync: {config_id}`

**Authentication:**
- `GitAuthConfig::Token` → HTTPS clone with token in URL or git credential helper
- `GitAuthConfig::SshKey` → SSH clone with key path passed to git2 callbacks

#### 1d. Auto-Sync on Save (the "commit on save" flow)

When `auto_sync: true` and `commit_strategy: PerSave`:

1. Frontend saves note via `useUpdateNote()` mutation
2. After successful Parachute update, check if note's path falls under any `DirectorySyncConfig`
3. If yes, invoke new Tauri command: `sync_push_file(note_id, config_id)`
4. Rust handler:
   - Write note content to the corresponding file in the local clone
   - `git add {file}`
   - `git commit -m "Update {filename}"`
   - `git push origin {branch}` (async, non-blocking)
5. Emit Tauri event `sync:pushed` for status bar notification

**Frontend integration point** — in `useUpdateNote()` onSuccess callback:
```typescript
onSuccess: (_, variables) => {
  // existing cache invalidation...
  // NEW: trigger auto-sync if configured
  checkAndTriggerSync(variables.noteId);
}
```

#### 1e. UI: Setting Up GitHub Sync

**Entry point:** Right-click directory in ProjectTree → "Sync to GitHub..."

**Dialog flow (modal):**
1. **Repository** — URL input (or "Create new on GitHub" which uses `gh` CLI)
2. **Branch** — default "main", allow custom
3. **Authentication** — GitHub token input (stored in macOS Keychain via `security-framework`)
4. **Commit strategy** — radio: Per Save / Batched / Manual
5. **Conflict strategy** — radio: Local Wins / Remote Wins / Ask
6. **Confirm** → runs initial sync (clone + push/pull)

**Status indicators:**
- ProjectTree: synced directories show a git icon badge
- StatusBar: last sync time + push/pull counts
- Sync errors surface as toast notifications

#### 1f. Handling Edge Cases

| Scenario | Behavior |
|----------|----------|
| Note has no file extension | Append `.md` (configurable) |
| Note path has spaces | Preserve as-is (git handles spaces fine) |
| Binary content in note | Skip sync, log warning |
| Remote has files not in vault | Pull → create new notes with appropriate path |
| Note deleted in vault | Delete file from repo on next sync |
| File deleted on GitHub | Delete note from vault on next sync (with confirmation if `conflict_strategy: "ask"`) |
| Auth token expired | Surface error, prompt re-auth |
| Push rejected (remote ahead) | Auto-pull + rebase, then retry push |
| Merge conflict in file | Use conflict strategy; for "ask", show diff UI |

#### 1g. Implementation Sequence

1. Add `git2` to Cargo.toml
2. Create `DirectorySyncConfig` model + `DirectorySyncAdapter` trait
3. Implement `GitHubAdapter` with clone/fetch/push/pull
4. Add Tauri commands: `github_sync_init`, `github_sync_push`, `github_sync_pull`, `github_sync_status`
5. Register commands in `lib.rs`
6. Add GitHub token to `AppConfig` + Keychain storage
7. Frontend: `GitHubSyncModal` component
8. Frontend: ProjectTree context menu "Sync to GitHub..."
9. Frontend: auto-sync hook in `useUpdateNote`
10. Frontend: sync status in StatusBar
11. Test with a real repo

## 2. Ollama + Prism MCP Client (Full Vault Access)

### The Problem

All AI operations currently go through Claude Code CLI (`claude -p`). Users want to:
- Use local models (Ollama) for certain skills to save cost/latency
- Keep Claude Code as the **primary** for everything (full MCP, session resumption, power)
- Have Ollama skills that **also** get full vault access via MCP — no second-class citizens

### Key Insight: MCP Is Model-Agnostic

Claude Code CLI is locked to Claude models, but the MCP protocol doesn't care what LLM calls it. The Parachute MCP server at `127.0.0.1:1940/mcp` just receives JSON-RPC tool calls and returns results. We can build a **Prism MCP Client** — a lightweight agent loop in Rust that:

1. Connects to the Parachute MCP server
2. Converts MCP tool definitions → Ollama's tool calling format
3. Runs the standard agent loop: prompt → tool calls → execute → feed back → repeat
4. Returns the final response

This means **Ollama gets the same vault access as Claude** — no MCP wall.

### Architecture: Two Paths, Same Power

```
┌─────────────────────────────────────────────────────┐
│                    ModelRouter                       │
│                                                      │
│  ┌────────────────────┐    ┌──────────────────────┐ │

│  │  (`claude -p`)      │    │  (built-in Rust)     │ │
│  │                     │    │                      │ │
│  │  ✓ Claude models    │    │  ✓ Ollama models     │ │
│  │  ✓ MCP via .mcp.json│    │  ✓ MCP via direct    │ │
```

```rust
/// Handles the JSON-RPC protocol: initialize, list tools, call tools.
pub struct PrismMcpClient {
    mcp_url: String,         // http://127.0.0.1:1940/mcp
    client: reqwest::Client,
    tools: Vec<McpToolDef>,  // cached tool definitions from server
}

pub struct McpToolDef {
    pub name: String,
    pub description: String,
    pub input_schema: serde_json::Value,  // JSON Schema for parameters

    /// Execute a tool call against the MCP server

        &self,
        name: &str,
        arguments: serde_json::Value,
    ) -> Result<serde_json::Value, PrismError>;
```

#### 2b. Ollama Client with Tool Calling

```rust
// src-tauri/src/clients/ollama.rs (new file)

/// Ollama HTTP client with tool-calling agent loop.
/// Connects to Ollama (localhost:11434) and Parachute MCP (localhost:1940/mcp).
pub struct OllamaAgent {
    ollama_url: String,
    client: reqwest::Client,
    mcp: PrismMcpClient,
    /// In-memory conversation history per context (mirrors AgentSessions pattern)
    conversations: Mutex<HashMap<String, Vec<OllamaChatMessage>>>,
}

#[derive(Serialize, Deserialize, Clone)]
pub struct OllamaChatMessage {
    pub role: String,         // "system" | "user" | "assistant" | "tool"
    pub content: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub tool_calls: Option<Vec<OllamaToolCall>>,
}

#[derive(Serialize, Deserialize, Clone)]
pub struct OllamaToolCall {
    pub function: OllamaFunctionCall,
}

#[derive(Serialize, Deserialize, Clone)]
pub struct OllamaFunctionCall {
    pub name: String,
    pub arguments: serde_json::Value,
}

impl OllamaAgent {
    pub fn new(ollama_url: String, mcp: PrismMcpClient) -> Self;

    /// The core agent loop:
    /// 1. Send prompt + MCP tools to Ollama /api/chat
    /// 2. If response contains tool_calls → execute each via MCP
    /// 3. Feed tool results back as "tool" role messages
    /// 4. Repeat until model produces a final text response (no tool calls)
    /// 5. Return final response
    ///
    /// Max iterations: 10 (prevent infinite loops)
    pub async fn run(
        &self,
        system_prompt: &str,
        user_prompt: &str,
        context_key: &str,   // for conversation history tracking
        model: &str,          // e.g. "llama3.3", "qwen2.5:72b"
        timeout_secs: u64,
    ) -> Result<String, PrismError> {
        let tools = self.mcp.tools_as_ollama_format();

        let mut messages = vec![
            OllamaChatMessage {
                role: "system".into(),
                content: system_prompt.into(),
                tool_calls: None,
            },
            OllamaChatMessage {
                role: "user".into(),
                content: user_prompt.into(),
                tool_calls: None,
            },
        ];

        // Prepend conversation history if resuming
        if let Some(history) = self.conversations.lock().unwrap().get(context_key) {
            let mut full = history.clone();
            full.extend(messages);
            messages = full;
        }

        for _iteration in 0..10 {
            let response = self.client
                .post(format!("{}/api/chat", self.ollama_url))
                .json(&serde_json::json!({
                    "model": model,
                    "messages": messages,
                    "tools": tools,
                    "stream": false,
                }))
                .send()
                .await?
                .json::<serde_json::Value>()
                .await?;

            let msg = &response["message"];
            let assistant_msg = OllamaChatMessage {
                role: "assistant".into(),
                content: msg["content"].as_str().unwrap_or("").into(),
                tool_calls: serde_json::from_value(
                    msg["tool_calls"].clone()
                ).ok(),
            };

            messages.push(assistant_msg.clone());

            // If no tool calls, we're done
            let tool_calls = match &assistant_msg.tool_calls {
                Some(tc) if !tc.is_empty() => tc.clone(),
                _ => {
                    // Save conversation history
                    self.conversations.lock().unwrap()
                        .insert(context_key.to_string(), messages);
                    return Ok(assistant_msg.content);
                }
            };

            // Execute each tool call via MCP
            for tc in &tool_calls {
                let result = self.mcp.call_tool(
                    &tc.function.name,
                    tc.function.arguments.clone(),
                ).await;

                messages.push(OllamaChatMessage {
                    role: "tool".into(),
                    content: match result {
                        Ok(v) => serde_json::to_string_pretty(&v).unwrap_or_default(),
                        Err(e) => format!("Tool error: {}", e),
                    },
                    tool_calls: None,
                });
            }
        }

        Err(PrismError::Agent("Ollama agent exceeded max iterations".into()))
    }

    /// List available Ollama models
    pub async fn list_models(&self) -> Result<Vec<ModelInfo>, PrismError>;

    /// Check Ollama is running
    pub async fn health(&self) -> Result<bool, PrismError>;
}
```

#### 2c. Model Router

```rust
// src-tauri/src/clients/model_router.rs (new file)

/// Routes AI requests to Claude Code CLI (default) or Ollama+MCP (opt-in).
/// Claude Code is the primary path. Ollama is a user-selected alternative
/// that still gets full MCP vault access via PrismMcpClient.
pub struct ModelRouter {
    claude: ClaudeClient,
    ollama: Option<OllamaAgent>,  // None if Ollama not configured/running
    skill_config: HashMap<String, SkillModelConfig>,
}

pub struct SkillModelConfig {
    pub provider: String,   // "claude" | "ollama"
    pub model: String,      // e.g. "sonnet" or "llama3.3"
}

impl ModelRouter {
    /// Route a request based on user's per-skill configuration.
    /// Default: Claude Code CLI for everything.
    pub async fn run(
        &self,
        skill: &str,            // "edit", "chat", "transform", "generate"
        system_prompt: &str,
        user_prompt: &str,
        context_key: &str,      // for session/conversation tracking
        timeout_secs: u64,
    ) -> Result<String, PrismError> {
        let config = self.skill_config.get(skill);

        match config.map(|c| c.provider.as_str()) {
            Some("ollama") => {
                let ollama = self.ollama.as_ref()
                    .ok_or(PrismError::Config("Ollama not available".into()))?;
                let model = &config.unwrap().model;
                ollama.run(system_prompt, user_prompt, context_key, model, timeout_secs).await
            }
            _ => {
                // Default: Claude Code CLI
                let model = config
                    .map(|c| c.model.as_str())
                    .unwrap_or("sonnet");
                let full_prompt = format!("{}\n\n{}", system_prompt, user_prompt);
                self.claude.run(&full_prompt, model, timeout_secs).await
            }
        }
    }

    /// Conversational route (for agent_chat with session resumption)
    pub async fn run_conversational(
        &self,
        skill: &str,
        system_prompt: &str,
        user_prompt: &str,
        context_key: &str,
        timeout_secs: u64,
    ) -> Result<LlmResponse, PrismError> {
        let config = self.skill_config.get(skill);

        match config.map(|c| c.provider.as_str()) {
            Some("ollama") => {
                let ollama = self.ollama.as_ref()
                    .ok_or(PrismError::Config("Ollama not available".into()))?;
                let model = &config.unwrap().model;
                let result = ollama.run(
                    system_prompt, user_prompt, context_key, model, timeout_secs
                ).await?;
                Ok(LlmResponse {
                    result,
                    session_id: Some(context_key.to_string()),
                    is_error: false,
                })
            }
            _ => {
                // Claude Code CLI with session resumption
                let session_id = /* get from AgentSessions */;
                let model = config.map(|c| c.model.as_str()).unwrap_or("sonnet");
                let full_prompt = format!("{}\n\n{}", system_prompt, user_prompt);
                let resp = self.claude.run_conversational(
                    &full_prompt, model, session_id, timeout_secs
                ).await?;
                Ok(LlmResponse {
                    result: resp.result,
                    session_id: resp.session_id,
                    is_error: resp.is_error,
                })
            }
        }
    }
}

pub struct LlmResponse {
    pub result: String,
    pub session_id: Option<String>,
    pub is_error: bool,
}
```

#### 2f. MCP Protocol Details

Parachute MCP uses **HTTP+SSE transport** (Streamable HTTP). The protocol flow:

```
1. POST /mcp  { jsonrpc: "2.0", method: "initialize", params: { ... } }
   → { capabilities: { tools: {} }, serverInfo: { ... } }

2. POST /mcp  { jsonrpc: "2.0", method: "tools/list" }
   → { tools: [ { name, description, inputSchema }, ... ] }

3. POST /mcp  { jsonrpc: "2.0", method: "tools/call", params: { name, arguments } }
   → { content: [ { type: "text", text: "..." } ] }
```

The PrismMcpClient implements this protocol directly over HTTP — no stdio, no subprocess. It's the same protocol Claude Code uses internally when it reads `.mcp.json`, but we own the client.

1. Create `PrismMcpClient` in `src-tauri/src/clients/mcp_client.rs` — JSON-RPC over HTTP
2. Create `OllamaAgent` in `src-tauri/src/clients/ollama.rs` — HTTP client + tool call loop
3. Create `ModelRouter` in `src-tauri/src/clients/model_router.rs` — routing logic

8. Add "AI Models" section to Settings UI
9. Test: Ollama + MCP tool calls against live Parachute vault

## 3. Notion Task Sync (Bidirectional)

### The Problem

The existing Notion sync adapter syncs **individual document content** (markdown ↔ Notion blocks). But task management is a different beast: Notion databases have **structured properties** (Status, Priority, Assignee, Due Date), and you want a Notion project board to stay in sync with Parachute task notes — bidirectionally.

This isn't document sync. It's **structured data sync between two different schema systems.**

### Key Insight: Property Mapping Is the Hard Part

A Notion database row has typed properties:
```json
{
  "Status": { "select": { "name": "In Progress" } },
  "Priority": { "select": { "name": "High" } },
  "Due": { "date": { "start": "2026-04-20" } },
  "Assignee": { "people": [{ "name": "Benjamin" }] },
  "Project": { "relation": [{ "id": "abc123" }] }
}
```

A Parachute task note has metadata:
```json
{
  "status": "in-progress",
  "priority": "high",
  "due": "2026-04-20",
  "assignee": "Benjamin",
  "project": "prism"
}
```

We need a **property map** that translates between these two schemas. And it needs to handle:
- Notion selects → Parachute string enums (with value normalization)
- Notion dates → ISO strings
- Notion people → plain strings
- Notion relations → Parachute links
- Notion rich text → Parachute markdown content
- Notion formulas/rollups → read-only metadata (can't push back)

### Architecture

#### 3a. New Adapter: `NotionDatabaseSyncAdapter`

This is fundamentally different from the existing `NotionAdapter` (document sync) and the `DirectorySyncAdapter` (GitHub). It syncs **a Notion database ↔ a set of tagged Parachute notes**.

```rust
// src-tauri/src/sync/adapters/notion_db.rs (new file)

pub struct NotionDatabaseAdapter {
    api_key: String,
    client: reqwest::Client,
}

/// Configuration for a Notion database ↔ Parachute tag sync
pub struct NotionDbSyncConfig {
    pub id: String,                          // UUID
    pub notion_database_id: String,          // Notion DB ID
    pub notion_database_name: String,        // Human-readable name
    pub parachute_tag: String,               // Tag to apply to synced notes (e.g. "task")
    pub parachute_path_prefix: String,       // Where to create notes (e.g. "vault/tasks/prism")
    pub property_map: Vec<PropertyMapping>,  // Notion prop → Parachute metadata field
    pub title_property: String,              // Which Notion property is the title (usually "Name")
    pub content_property: Option<String>,    // Which Notion prop maps to note content (if any)
    pub sync_direction: String,              // "bidirectional" | "notion-to-parachute" | "parachute-to-notion"
    pub conflict_strategy: String,           // "notion-wins" | "parachute-wins" | "newer-wins"
    pub last_synced: String,                 // ISO timestamp
    pub auto_sync: bool,
    /// Maps Notion page IDs ↔ Parachute note IDs for identity tracking
    pub id_map: HashMap<String, String>,     // notion_page_id → parachute_note_id
}

pub struct PropertyMapping {
    pub notion_property: String,     // e.g. "Status"
    pub notion_type: NotionPropType, // select, date, people, rich_text, etc.
    pub parachute_field: String,     // e.g. "status"
    pub transform: ValueTransform,   // how to convert between the two
}

pub enum NotionPropType {
    Title,
    RichText,
    Select,
    MultiSelect,
    Date,
    People,
    Checkbox,
    Number,
    Url,
    Relation,
    Formula,    // read-only from Notion
    Rollup,     // read-only from Notion
}

pub enum ValueTransform {
    /// Direct pass-through (string → string)
    Identity,
    /// Normalize: lowercase, replace spaces with hyphens
    /// e.g. "In Progress" → "in-progress"
    Slugify,
    /// Map specific values: { "To Do": "todo", "In Progress": "active", "Done": "complete" }
    ValueMap(HashMap<String, String>),
    /// Extract first date from Notion date object
    DateExtract,
    /// Extract person names as comma-separated string
    PeopleExtract,
    /// Notion relation IDs → Parachute links
    RelationToLinks { relationship_type: String },
}
```

#### 3b. Sync Flow

**Notion → Parachute (pull):**
```
1. Query Notion database: POST /v1/databases/{id}/query
   - Filter: last_edited_time > last_synced (or full sync if first run)
   - Paginate (100 per page)
2. For each Notion page:
   a. Check id_map: does this page already have a Parachute note?
   b. If yes → UPDATE:
      - Extract properties using property_map
      - Convert to Parachute metadata fields
      - Update note content (from page body blocks → markdown)
      - Update metadata via Parachute API
   c. If no → CREATE:
      - Create new note at {path_prefix}/{slugified-title}
      - Apply parachute_tag
      - Set metadata from mapped properties
      - Add to id_map
3. Check for deletions:
   - For each entry in id_map, check if Notion page still exists
   - If deleted from Notion → flag for deletion (or archive) in Parachute
```

**Parachute → Notion (push):**
```
1. Query Parachute notes by tag: GET /api/notes?tag={parachute_tag}&path={path_prefix}
   - Filter: updatedAt > last_synced
2. For each modified note:
   a. Check id_map: does this note have a Notion page?
   b. If yes → UPDATE:
      - Reverse-map metadata fields to Notion properties
      - PATCH /v1/pages/{id} with updated properties
      - Update page content (markdown → Notion blocks)
   c. If no → CREATE:
      - POST /v1/pages with parent: { database_id }
      - Set properties from metadata
      - Set page body from note content
      - Add to id_map
3. Handle deletions:
   - Notes deleted from Parachute → archive Notion page (not delete — safer)
```

**Bidirectional conflict resolution:**
```
Both changed since last sync?
- "newer-wins": compare updatedAt timestamps
- "notion-wins": always pull from Notion
- "parachute-wins": always push to Parachute
- Future: "merge" — merge non-conflicting fields, flag conflicting ones
```

#### 3c. Auto-Discovery of Property Mappings

When a user connects a Notion database, we can **auto-generate the property map** by reading the database schema:

```
1. GET /v1/databases/{id} → returns { properties: { ... } }
2. For each property:
   - Match by name to known Parachute metadata fields:
     "Status" → status, "Priority" → priority, "Due" / "Due Date" → due,
     "Assignee" → assignee, "Project" → project, "URL" → url
   - Infer transform from type:
     select → Slugify, date → DateExtract, people → PeopleExtract
   - User can customize/override in the setup UI
3. Show preview: "We'll map these properties:"
   Status (select) → status (slugify)
   Priority (select) → priority (slugify)
   Due Date (date) → due
   Assignee (people) → assignee
   [Edit mappings...]
```

#### 3d. Notion Relations → Parachute Links

This is where it gets powerful. Notion databases can have **relation properties** that link to other databases. For example:

- Task database has "Project" relation → Project database
- Task database has "Blocked By" relation → same Task database

When we sync:
1. If the related Notion page is also synced to Parachute → create a Parachute link
2. Relationship type comes from the property name (e.g. "Blocked By" → `"blocked-by"`)
3. If the related page isn't synced yet → store the Notion ID, resolve lazily on next sync

```rust
// In the sync logic:
if let ValueTransform::RelationToLinks { relationship_type } = &mapping.transform {
    for related_notion_id in notion_relation_ids {
        if let Some(parachute_note_id) = id_map.get(&related_notion_id) {
            // Create link in Parachute
            parachute.update_note(&note_id, &UpdateNoteParams {
                links_add: Some(vec![LinkAdd {
                    target: parachute_note_id.clone(),
                    relationship: relationship_type.clone(),
                }]),
                ..Default::default()
            }).await?;
        } else {
            // Related page not yet synced — queue for deferred resolution
            deferred_links.push((note_id.clone(), related_notion_id, relationship_type.clone()));
        }
    }
}
```

   - Recommendation: In-memory HashMap (like existing `AgentSessions`), don't persist — conversations are ephemeral

   - Recommendation: Full vault with optional path/tag filters — matches existing search UX

5. **Notion task sync granularity:** Sync entire database or allow filtering by Notion view/filter?
   - Recommendation: Start with entire database. Add view-based filtering later if databases get large.

6. **Notion sync frequency:** How often to poll?
   - Recommendation: 5 minutes default (configurable). Notion's API doesn't support webhooks for database changes, so polling is the only option. On explicit user action (save, drag task), push immediately.

7. **MCP transport for PrismMcpClient:** Parachute uses Streamable HTTP. Should we also support stdio transport for future MCP servers?
   - Recommendation: HTTP only for now. The PrismMcpClient is purpose-built for Parachute. If we need other MCP servers later, extend then.

8. **Notion property mapping persistence:** Where to store the auto-discovered + user-customized property map?
   - Recommendation: In `~/.config/prism/notion-sync/{database_id}.json`. The mapping is config, not vault data — it shouldn't live in Parachute.
