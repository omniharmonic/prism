# Prism v0.1.2 — Architecture Plan

**Date:** April 12, 2026 | **Status:** Planning
**Theme:** Parachute as canonical data layer, agent-native operations, unified personal OS

---

## Design Philosophy

Prism is not a UI that reads from external services. It is a **personal operating system** where:

- **Parachute** is the kernel — the single source of truth for all data
- **The agent** is the co-pilot — continuously enriching, triaging, and managing the vault
- **External services** (Matrix, Google, Notion) are I/O devices — data flows in, gets enriched, flows back out
- **Person** is the universal join key — every message, meeting, task, and document links back to person notes

The delineation between Prism and the agent: **Prism is the interface, the agent is the intelligence.** The agent reads everything, enriches everything, and surfaces what matters. Prism renders it beautifully and lets you act on it.

## 1. Parachute as Canonical Data Store

### Problem
Currently, messages live in Matrix, calendar in Google, and only documents live in Parachute. This means search, linking, and agent processing can't cross boundaries. The agent can't say "the email from Patricia mentions the same project as yesterday's meeting" because they're in different systems.

### Design: Universal Ingestion Pipeline

Everything flows into Parachute as tagged notes with rich metadata and links:

```
External Service → Ingestion Adapter → Parachute Note (tagged) → Agent Enrichment → Links Created
```

#### 1a. Message Ingestion (Matrix → Parachute)

**Background sync service** runs every 60 seconds:
1. Poll Matrix `/sync` for new events since last token
2. For each new message in a bridged room:
   - Find or create the **conversation note** (`vault/messages/{platform}/{contact-name}`)
   - Tag: `message-thread`, `{platform}` (e.g., `whatsapp`)
   - Append the new message to the note's content
   - Link the conversation note to the **person note** via `"messages-with"` relationship
   - Store metadata: `{ platform, matrixRoomId, lastMessageAt, participants[], unreadCount }`
3. For DMs: link directly to person note
4. For groups: link to each participant's person note

**Message format in Parachute note:**
```markdown
# Patricia Parkinson — WhatsApp

[2026-04-12 09:15] Patricia: Hey, are you available for a call about the BGI proposal?
[2026-04-12 09:17] Benjamin: Yes, let me wrap up what I'm doing. 15 minutes?
[2026-04-12 09:17] Patricia: Perfect, I'll send the Zoom link
```

**Agent enrichment** (runs after ingestion):
- Classify importance: `urgent` | `action-required` | `informational` | `social`
- Extract action items → create linked task notes
- Detect mentions of projects/people → create links
- Tag with topic: `proposal`, `scheduling`, `follow-up`, etc.

#### 1b. Calendar Ingestion (Google → Parachute)

**Sync every 5 minutes:**
1. Fetch events for the next 7 days + past 1 day
2. For each event, find or create a **meeting note** (`vault/meetings/{date}/{slug}`)
3. Tag: `meeting` (or `event` for non-meeting calendar items)
4. Metadata: `{ date, start, end, attendees[], location, meetLink, calendarEventId, status }`
5. Link to attendee person notes
6. Content: starts empty, populated by agent after meeting (transcript integration)

**Bidirectional sync:**
- Editing event metadata in Prism → pushes back to Google Calendar via gog
- Creating a meeting note in Prism with the right metadata → creates Google Calendar event
- Deleting → deletes in both

#### 1c. Email Ingestion (Gmail → Parachute)

**Sync every 3 minutes** (mirrors current omniharmonic email triage pattern):
1. Fetch recent threads via `gog gmail messages search "in:inbox newer_than:3h"`
2. For each thread, find or create an **email note** (`vault/messages/email/{subject-slug}`)
3. Tag: `email`
4. Metadata: `{ from, to[], cc[], subject, threadId, labels[], importance }`
5. Link to person notes for sender and recipients
6. Content: full email thread text

**Agent triage** (same as current omniharmonic pattern but writes to Parachute):
- URGENT → tag `urgent`, create task, Telegram alert
- ACTION → tag `action-required`, create linked task
- INFORMATIONAL → tag `informational`
- Low priority → tag `social` or `newsletter`

## 2. Person-Centric Data Model

### The Person Note as Universal Hub

Every person in the vault becomes a hub node in the knowledge graph:

```
                    ┌─── WhatsApp messages ──── [conversation note]
                    ├─── Telegram messages ──── [conversation note]
                    ├─── Emails ─────────────── [email thread note]
[Person: Patricia] ─┤
                    ├─── Meetings attended ──── [meeting note]
                    ├─── Tasks assigned ──────── [task note]
                    ├─── Projects shared ──────── [project note]
                    └─── Documents mentioned ── [document note]
```

**Person note metadata:**
```json
{
  "type": "person",
  "name": "Patricia Parkinson",
  "channels": {
    "whatsapp": "+1234567890",
    "telegram": "@patricia_p",
    "email": ["patricia@opencivics.com"],
    "matrix": "@patricia:localhost"
  },
  "matrixRoomIds": {
    "whatsapp": "!abc123:localhost",
    "telegram": "!def456:localhost"
  },
  "organizations": ["OpenCivics", "BGI"],
  "role": "Co-Founder, OpenCivics"
}
```

**Unified messaging UX:**
1. User types a person's name
2. Prism shows all channels for that person
3. User picks a channel (or lets the agent suggest based on context)
4. Message sends via the appropriate Matrix bridge room
5. Response comes back through Matrix → ingested to Parachute → appears in Prism

**"Show me everything about Patricia"** = graph traversal from person note:
- All conversations (across platforms)
- All meetings attended together
- All shared projects
- All tasks involving them
- All documents that mention them

## 3. Agent-Native Architecture

### Problem
The current agent can generate text but can't directly edit documents, and the experience doesn't feel like real-time collaboration. The omniharmonic agent has powerful skills (briefing, triage, intelligence) that don't integrate with Prism.

### Design: Three Agent Modes

#### 3a. Inline Editor (Real-time Collaboration)

**For document editing — feels like pair programming.**

**Current state (v0.1.1, working):** Claude Code CLI with MCP access. Agent uses
`mcp__parachute-vault__update-note` to edit the document directly. Prism detects the
change, shows ghost text overlay with Accept/Reject. The "Edit" toggle in PanelChat
controls whether responses auto-trigger the diff view.

**v0.1.2 refinements to ship:**

1. **Smarter diff display** — Instead of showing the entire new document as ghost text,
   compute an actual diff between old and new content. Highlight added/removed/changed
   sections inline in the editor (green for additions, red strikethrough for deletions).
   This gives a true "track changes" experience.

2. **Partial accept** — Allow accepting individual changes, not just all-or-nothing.
   Each diff hunk gets its own Accept/Reject button. This is critical for longer documents
   where the agent makes multiple changes.

3. **Edit context preservation** — Before the agent edits, snapshot the editor state
   (cursor position, scroll, selection). After Accept, restore position. After Reject,
   ensure the editor state is unchanged.

4. **Loading indicator during edit** — Show a "Claude is editing..." indicator in the
   editor area while the agent is running. Currently there's no visual feedback between
   clicking send and the ghost text appearing.

5. **Multi-note awareness** — When the agent edits the current note, it might also
   create linked notes (tasks, meeting notes) or add tags. Surface these secondary
   changes in a "Related changes" section below the diff.

6. **Undo after accept** — Store the pre-accept content so the user can undo
   (Cmd+Z) back to their version even after accepting.

7. **Auto-edit detection** — Instead of requiring "Edit" toggle, detect when the
   agent used MCP to update the current note and always show the diff. The toggle
   can control whether to auto-insert the chat response text too.

#### 3b. Background Agent (Continuous Enrichment)

**For automated processing — runs continuously:**

Migrate omniharmonic agent skills to Prism-native background services:

1. **Message Triage** (replaces email triage agent)
   - Runs on every new message ingested to Parachute
   - Classifies importance, extracts action items
   - Tags notes, creates tasks, sends alerts

2. **Meeting Processor** (replaces briefing agent's transcript handling)
   - When a meeting note gets a transcript (from Meetily or manual upload)
   - Extracts attendees, action items, decisions, topics
   - Creates linked task notes, updates person notes

3. **Intelligence Engine** (replaces intelligence.py)
   - Runs daily (or on-demand)
   - TaskEscalationAnalyzer: graduated urgency for overdue tasks
   - SlotOpportunityDetector: match calendar gaps to tasks
   - CommitmentTracker: things others committed to
   - PatternDetector: stalled projects, meeting density
   - Writes insights to `vault/agent/insights/{date}` tagged `agent-insight`

4. **Daily Briefing** (replaces briefing agent)
   - Generates morning briefing from vault state
   - Surfaces: escalations, overdue tasks, today's schedule, follow-ups
   - Writes to `vault/agent/briefings/{date}` tagged `briefing`
   - Can also send to Telegram

#### 3c. On-Demand Agent (Task Dispatch)

**For complex work — mirrors omniharmonic dispatch:**

The dispatch pattern from the omniharmonic dashboard, but native in Prism:
1. User creates a task and clicks "Dispatch to Agent"
2. Prism spawns a Claude Code process with full MCP access
3. Agent reads task context, plans work, executes
4. Progress reported to an "Agent Activity" panel in Prism
5. On completion: results written to vault, task updated, notification sent

**Agent Activity Dashboard** (new tab in Prism):
- Shows active dispatches with progress
- Completed dispatches with output summaries
- Agent insights and suggestions
- Triage queue (messages/emails awaiting review)

### Design

3. **Edit event:**
   - Change any field → syncs back to Google Calendar

4. **Meeting notes integration:**
   - Each event can have an associated note
   - Click "Open Notes" → creates or opens `vault/meetings/{date}/{slug}`
   - After meeting: paste transcript, agent processes it

## 6. Skills & Plugin Architecture

### Design: Agent Skills as Parachute-Native Operations

Each skill is a prompt template + tool configuration that the agent can execute:

```yaml
# vault/agent/skills/email-triage.yaml
name: email-triage
trigger: schedule  # or "on-ingest", "manual"
schedule: "*/3 * * * *"  # every 3 hours
prompt: |
  Triage the following emails. For each:
  - Classify: URGENT, ACTION, INFORMATIONAL, SOCIAL
  - Extract action items as tasks
  - Link to relevant person and project notes
tools:
  - mcp__parachute-vault__search-notes
  - mcp__parachute-vault__create-note
  - mcp__parachute-vault__tag-note
  - mcp__parachute-vault__get-links
context:
  - active-tasks
  - recent-meetings
  - person-directory
```

Skills stored as Parachute notes tagged `agent-skill`. The agent reads its own skills from the vault, making them user-editable and versionable.

## 8. Architecture Decisions

### Streaming: API vs CLI
- **Anthropic API** for editing (streaming tokens, tool_use for structured ops)
- **Claude Code CLI** for background tasks (MCP access, session resumption, complex multi-step)
- Store API key in macOS keychain (already implemented: `try_keychain_anthropic()`)

### Message Storage: Matrix vs Parachute
- **Parachute is canonical** — all messages indexed as notes
- **Matrix is the transport** — handles real-time delivery, bridge protocols, encryption
- **Sync is one-way for content** (Matrix → Parachute) but **two-way for actions** (send message → Matrix → bridges → platform)

### Background Services: Rust vs Agent
- **Rust background tasks** for deterministic operations: sync polling, message ingestion, calendar sync
- **Agent (Claude) tasks** for intelligence: triage, extraction, insights, briefings
- Agent tasks triggered by Rust when new data arrives

### Person Matching: Automated vs Manual
- Start with **automated fuzzy matching** (Matrix display name → Parachute person note name)
- Allow **manual corrections** via person note metadata
- Store canonical mappings in person note `channels` and `matrixRoomIds` fields

## Appendix A: Real-Time Collaboration & Sharing (v0.1.3 Stretch Goal)

> **Status:** Conceptual design — pending Parachute API primitives from Aaron Gabriel.
> This section models the user flows, permissions architecture, and integration
> points so Prism is ready to build on those primitives when they land.

### A1. The Vision

Prism today is a single-player experience. v0.1.3 makes it multiplayer:

- **Auto-sync**: Notes stay current with their sync sources without manual push/pull
- **Collaborative workspaces**: Share project-scoped slices of your vault with collaborators
- **Web viewer**: Anyone with a link can read shared content (no Prism/Parachute required)
- **Cross-vault federation**: Collaborators with their own Parachute can sync shared notes bidirectionally
- **Agent interop**: A collaborator's agent can query your shared notes via a scoped API

The key insight: **sharing is a filtered, permissioned view of the vault graph — not a separate system.** Tags and links already organize everything; collaboration adds access control and transport.

### A2. Auto-Sync Enhancement

#### Problem
Currently, sync is manual (push/pull buttons in the UI). Users have to remember to sync, and stale content causes confusion.

#### Design: Sync-on-Access + Sync-on-Save

**On note open:**
1. Prism checks if the note has a sync source (Google Doc, Notion page, etc.)
2. If auto-sync is enabled, triggers a background pull before rendering
3. Shows a brief "Syncing..." indicator, then renders the latest content
4. If the pull fails (offline, source deleted), renders cached version with a stale indicator

**On save:**
1. After auto-save writes to Parachute, check if the note has a sync target
2. If auto-sync is enabled, push changes to the external source
3. If push fails, queue for retry and show a sync-pending indicator

**Settings (per-source or global):**
```
Sync Mode:
  ○ Manual — push/pull only when you click the button
  ○ On-open — pull latest when you open a note; push on save
  ○ Continuous — background poll every N seconds (for actively edited shared docs)
```

**Conflict resolution:**
- If both local and remote changed since last sync: show a merge dialog
- For Parachute-to-Parachute sync (v0.1.3): use vector clocks or CRDTs (depends on Aaron's primitives)
- For external sources (Google Docs, Notion): last-write-wins with the option to view the diff

#### Implementation Notes
- Extend `src-tauri/src/sync/` adapters with `pull_if_stale()` and `push_on_save()` methods
- Add `sync_mode` field to note metadata: `"manual"` | `"on-open"` | `"continuous"`
- Continuous mode: Rust background task per active note, polls at configurable interval
- Sync status tracked in Zustand store: `{ noteId, status: "synced" | "syncing" | "stale" | "conflict" }`

### A3. Collaborative Workspaces

#### Concept: Project-Scoped Sharing

A **workspace** is a sharing scope defined by a tag query:

```
Share: everything tagged "project:bioregional" with Patricia
Permission: can-edit
Includes: 47 notes (documents, tasks, meetings, research)
```

This means:
- All notes matching the tag filter are visible to the collaborator
- New notes that match the filter are automatically included
- The collaborator sees a consistent, curated slice of your vault

#### User Flow: Sharing a Workspace

1. **Initiate sharing:**
   - Right-click a project in the sidebar → "Share workspace..."
   - Or: Settings → Workspaces → "Create shared workspace"
   
2. **Configure scope:**
   - Tag filter: `project:bioregional` (or any tag query)
   - Optional path filter: `vault/projects/bioregional/**`
   - Optional exclusions: `tag:personal`, `tag:draft`
   
3. **Set permissions per collaborator:**
   - **View** — read-only access to matching notes
   - **Suggest** — can propose edits (appear as ghost text for the owner to accept/reject)
   - **Edit** — full read-write (changes sync bidirectionally)
   - **Admin** — can add other collaborators, change scope
   
4. **Choose distribution:**
   - 🔗 **Web link** — generates a read-only URL (no account needed)
   - 🔄 **Vault sync** — syncs to collaborator's Parachute instance
   - 📁 **Filesystem export** — syncs to a local directory on collaborator's machine
   - 🤖 **Agent API** — exposes an MCP endpoint for the collaborator's agent

5. **Collaborator experience:**
   - Receives an invite (via email, Matrix message, or shared link)
   - If they have Prism: workspace appears in their sidebar as a "Shared with me" section
   - If they don't: web viewer shows the content

#### Workspace Data Model

```yaml
# Stored as a Parachute note tagged "workspace"
id: ws_bioregional_patricia
name: "Bioregional Project — Patricia"
owner: benjamin
collaborators:
  - person: patricia  # links to person note
    permission: edit
    distribution: vault-sync
    vault_endpoint: "https://patricia.parachute.dev/api"
  - person: aaron
    permission: view
    distribution: web-link
scope:
  include_tags: ["project:bioregional"]
  exclude_tags: ["personal", "draft"]
  include_paths: []  # empty = all paths
created: 2026-05-01
sync_token: "ws_abc123_v42"  # tracks what's been synced
```

### A4. Distribution Channels

#### A4a. Web Viewer

**For casual sharing — no account required.**

A lightweight web app (or Vercel-deployed static site) that renders Parachute notes:

- Owner generates a share link: `https://prism.share/ws/abc123` (or self-hosted)
- Link resolves to a read-only view of the workspace
- Renders markdown/HTML content with the glass design system
- Navigation: sidebar with note tree, search within workspace
- Real-time: polls for updates (or WebSocket if Parachute supports it)
- Optional: password protection, expiration, watermarking

**Architecture:**
```
Prism → Parachute API (scoped token) → Share Proxy → Web Viewer (React SPA)
```

The share proxy is a thin server that:
1. Validates the workspace token
2. Queries Parachute with the workspace scope filter
3. Returns only notes matching the scope + permission level
4. Never exposes the full vault

#### A4b. Vault-to-Vault Sync (Federation)

**For collaborators who run their own Parachute.**

This is the most powerful mode — full bidirectional sync between two Parachute instances:

1. Owner's Parachute exposes a **federation endpoint**: `POST /api/federation/sync`
2. Collaborator's Parachute periodically pulls updates from the owner's endpoint
3. Changes flow both ways (if permission allows)
4. Conflict resolution via vector clocks or CRDT merging

**Sync protocol (conceptual):**
```
1. Collaborator requests: GET /api/federation/{workspace_id}/changes?since={token}
2. Owner returns: { notes: [...changed notes], deleted: [...ids], new_token: "v43" }
3. Collaborator applies changes to their local vault (in a namespaced path)
4. If edit permission: collaborator pushes their changes back via POST
5. Owner applies changes, notifies Prism of updates
```

**Namespacing:** Federated notes live under `vault/shared/{workspace-name}/` in the collaborator's vault to avoid path conflicts with their own notes.

**Dependency:** This heavily depends on Aaron's Parachute API work. The federation endpoint, scoped tokens, and change-tracking are core primitives we'll need.

#### A4c. Filesystem Export

**For collaborators who want plain files.**

- Shared notes sync to a local directory as markdown files
- Folder structure mirrors vault paths within the workspace scope
- One-way (owner → collaborator) unless edit permission, then bidirectional via file watcher
- Useful for: giving collaborators access via their preferred editor, git integration

**Implementation:** Extend the existing sync adapter pattern — `FilesystemSyncAdapter` that writes `.md` files and watches for changes.

#### A4d. Agent API (MCP Endpoint)

**For collaborators whose agents need programmatic access.**

Expose a scoped MCP server that mirrors Parachute's MCP tools but filtered to the workspace:

```json
{
  "mcpServers": {
    "benjamin-bioregional": {
      "url": "https://benjamin.prism.dev/mcp/ws_bioregional",
      "token": "scoped_token_here"
    }
  }
}
```

A collaborator's Claude Code (or any MCP client) can then:
- `search-notes` — only returns notes in the workspace scope
- `get-note` — only if the note is in scope
- `create-note` / `update-note` — only if they have edit permission
- `traverse-links` — follows links but stops at the workspace boundary

**This is the most novel feature**: it means Patricia's agent can research Benjamin's project notes, draft responses referencing shared documents, and even create task notes — all within the permissioned workspace boundary.

### A5. Permissions & Trust Model

#### Permission Levels

| Level | Read | Suggest | Write | Share | Manage |
|-------|------|---------|-------|-------|--------|
| **View** | ✓ | — | — | — | — |
| **Suggest** | ✓ | ✓ | — | — | — |
| **Edit** | ✓ | ✓ | ✓ | — | — |
| **Admin** | ✓ | ✓ | ✓ | ✓ | ✓ |

#### Suggest Mode (Ghost Text for Collaboration)

The "Suggest" permission reuses the ghost text infrastructure from agent editing:
1. Collaborator edits a note in their Prism
2. Instead of writing directly to the owner's vault, the change is sent as a **suggestion**
3. Owner sees it as ghost text with Accept/Reject — identical UX to agent edits
4. Accepted suggestions become real changes; rejected ones are discarded

This is a natural extension of v0.1.2's agent editing UX — the same Accept/Reject flow works for both AI suggestions and human suggestions.

#### Trust Boundaries

```
┌─────────────────────────────────────────────┐
│  Your Vault (full access)                   │
│                                             │
│  ┌─────────────────────────────────┐        │
│  │  Workspace A (scoped view)     │ ← Patricia (edit)
│  │  - tagged: project:bioregional │ ← Aaron (view)
│  │  - excludes: personal, draft   │ ← Patricia's agent (read-only MCP)
│  └─────────────────────────────────┘        │
│                                             │
│  ┌─────────────────────────────────┐        │
│  │  Workspace B (scoped view)     │ ← Jim (suggest)
│  │  - tagged: project:opencivics  │
│  └─────────────────────────────────┘        │
│                                             │
│  [Everything else: private]                 │
└─────────────────────────────────────────────┘
```

Key principles:
- **Default private**: Nothing is shared unless explicitly added to a workspace
- **Scope is a filter, not a copy**: If you remove a tag from a note, it leaves the workspace
- **Audit trail**: All workspace access is logged (who accessed what, when)
- **Revocation is instant**: Remove a collaborator → their access ends immediately, cached content expires
- **Agent access is scoped**: An external agent can never traverse beyond the workspace boundary

### A7. Dependencies on Parachute API

The following primitives are needed from Parachute (Aaron's roadmap):

| Primitive | Purpose | Prism Readiness |
|-----------|---------|-----------------|
| **Scoped API tokens** | Token that can only access notes matching a filter | Need for all sharing |
| **Change feed / webhooks** | "What changed since token X?" for sync | Need for vault-to-vault sync |
| **Federation endpoint** | Cross-instance sync protocol | Need for vault-to-vault sync |
| **Access control lists** | Per-note or per-scope permissions | Can approximate with scoped tokens initially |
| **Suggestion queue** | Store pending suggestions separately from content | Need for suggest mode (can build in Prism initially) |
| **Audit log** | Track access to shared notes | Nice-to-have, can build in Prism |

**What Prism can build now (before Aaron's primitives):**
- Auto-sync on-open/on-save (just extends existing sync adapters)
- Workspace UI (create, configure, manage — stores as vault notes)
- Web viewer (read-only, using existing Parachute HTTP API + scoped queries)
- Ghost text for suggestions (reuses agent editing infrastructure)
- Permission model design (enforced client-side initially, server-side when ACLs land)

**What needs Parachute primitives:**
- Vault-to-vault federation (needs change feed + federation endpoint)
- Scoped MCP for agent interop (needs scoped tokens)
- Server-enforced permissions (needs ACLs)
- Real-time collaborative editing (needs WebSocket or CRDT support)
