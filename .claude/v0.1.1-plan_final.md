## 1. Live Agent Editing

### Problem
The Claude Code agent can generate content but can't directly edit the open document in real-time. The user has to click "Apply" or "Replace" after the agent responds. The agent also can't pull rich context about the current document (tags, links, metadata) without MCP tools that are slow to initialize.

### Design

**Approach: Streaming agent edits with live cursor**

Instead of the current batch pattern (agent generates → user applies), implement streaming where the user can watch the agent type into the document:

1. **Rich context injection** — When a chat session starts, the Rust backend pre-fetches everything about the current note:
   - Note ID, path, tags, metadata
   - Tag schema fields and values
   - Linked notes (from Parachute links API)
   - Recent edit history
   - All of this injected into the Claude prompt as structured context

2. **Streaming response → live editor updates** — Instead of `claude -p` (which waits for full completion), use `claude -p --output-format stream-json` to get token-by-token output. As tokens arrive:
   - Stream them into a "ghost text" overlay in the TipTap editor
   - Ghost text shows in a dimmed accent color
   - When streaming completes, show Accept/Reject buttons
   - Accept commits the text; Reject removes it

3. **Direct vault operations** — For operations that don't need the editor (like "tag this note as important", "create a new task from this paragraph"), the agent response includes structured commands that the Rust backend executes directly:
   ```json
   { "action": "tag", "noteId": "...", "tags": ["important"] }
   { "action": "create-note", "content": "...", "tags": ["task"] }
   { "action": "create-link", "sourceId": "...", "targetId": "..." }
   ```

### Key Files
- `src-tauri/src/commands/agent.rs` — streaming response handler
- `src/components/agent/PanelChat.tsx` — streaming UI + ghost text
- `src/components/renderers/DocumentRenderer.tsx` — ghost text overlay in TipTap
- `src/app/stores/ui.ts` — streaming state management

### Open Questions
- Should the agent be able to edit while the user is also typing? (probably not — show a "Claude is editing" lock)
- Should we use Claude Code CLI streaming or the Anthropic API directly for lower latency?
- How to handle multi-file edits (agent wants to update the current doc AND create a related task)?

## 2. Unified Communications Hub

### Problem
The inbox is one flat list of Matrix rooms. There's no way to:
- See individual threads within a conversation
- Open a specific platform's messages
- Send a message to a person across any platform
- View a unified contact profile

### Design

**2a. Inbox redesign — grouped by source**

```
┌─────────────────────────────────────┐
│ INBOX                               │
│                                     │
│ ▾ WhatsApp (3 unread)               │
│   Patricia Parkinson  — Hey, about… │
│   Aaron Gabriel       — See you to… │
│   OpenCivics Group    — Meeting at… │
│                                     │
│ ▾ Telegram (5 unread)               │
│   #General - Treegens — New propos… │
│   4D International    — Update on … │
│   Regen Hub           — Next event… │
│                                     │
│ ▾ Email (2 unread)                  │
│   jim@example.com     — Re: Propos… │
│   newsletter@sub.com  — Weekly dig… │
│                                     │
│ ▾ Discord (1 unread)                │
│   #opencivics-dev     — PR merged:… │
└─────────────────────────────────────┘
```

Click a thread → opens in the canvas as a MessageRenderer. Click the source header → opens a dashboard for that platform.

**2b. Contact-centric messaging**

Each person in the vault (`person` tag) has contact info across platforms. When composing a new message:

1. Type the person's name (autocomplete from vault `person` notes)
2. See all their communication channels: WhatsApp, Telegram, Email, Discord, etc.
3. Pick the channel
4. Compose and send

This requires linking Matrix room members to Parachute `person` notes. The bridge user IDs (`@whatsapp_1234:localhost`) need to be mapped to `vault/people/Name`.

**2c. Messages dashboard**

A dedicated "Messages" view (not just a sidebar section) with:
- All conversations across all platforms in one timeline
- Filter by platform, by person, by importance tag
- Search across all message history
- Pinned/starred conversations

**2d. Message indexing in Parachute**

Currently, messages live in Matrix's PostgreSQL database. To make them searchable and linkable in the knowledge graph:

- **Option A: Index messages as Parachute notes** — A background service periodically reads Matrix rooms and creates/updates Parachute notes tagged `message-thread` for each conversation. Content = last N messages. Metadata includes participants, platform, unread count.
- **Option B: Query Matrix directly** — Keep messages in Matrix, but create `person` note links to Matrix room IDs. When viewing a person, query their Matrix rooms for recent messages.
- **Recommended: Hybrid** — Index important messages (flagged by agent) as Parachute notes. Keep the full history in Matrix. Link both via person notes.

**2e. Importance tagging**

The OmniHarmonic agent (or a new Prism background service) monitors incoming Matrix messages and tags them:
- `urgent` — needs immediate response
- `action-required` — needs a response but not immediately  
- `informational` — FYI, no response needed
- `social` — casual conversation

These tags appear as badges in the inbox. The Messages dashboard can filter by importance.

### Key Files
- `src/components/navigation/Inbox.tsx` — redesign with grouped sources
- `src/components/comms/MessagesDashboard.tsx` — new full-page messages view
- `src/components/comms/ComposeMessage.tsx` — new composition with person picker
- `src/components/comms/ContactCard.tsx` — person profile with all channels
- `src-tauri/src/services/message_indexer.rs` — background Matrix → Parachute sync
- `src/lib/matrix/contact-map.ts` — Matrix user ID → Parachute person mapping

### Data Model
```typescript
// Person note metadata (existing, extended)
interface PersonMeta {
  type: "person";
  name: string;
  // Communication channels
  channels?: {
    whatsapp?: string;      // phone number or Matrix bridge ID
    telegram?: string;      // username
    discord?: string;       // username#discriminator
    email?: string[];       // email addresses
    matrix?: string;        // Matrix user ID
  };
  // Matrix room IDs for DMs with this person
  matrixRooms?: string[];
}

// Message thread note (indexed from Matrix)
interface MessageThreadMeta {
  type: "message-thread";
  platform: string;
  participants: string[];   // Parachute person note IDs
  matrixRoomId: string;
  importance?: "urgent" | "action-required" | "informational" | "social";
  lastMessageAt: string;
  unreadCount: number;
}
```

## 3. Vault Structure Management

### Problem
- The sidebar shows the vault's directory structure, but users can't create, rename, or reorganize directories
- The top-level "Projects" label is hardcoded — not everyone organizes by projects
- New notes are created with flat paths like "Untitled document 2026-04-11-..."
- No way to move notes between directories

### Design

**3a. Directory operations in the sidebar**

Right-click context menu on any folder or note:
- **New folder** — creates a directory (just a path prefix in Parachute)
- **Rename** — changes the note's path
- **Move to...** — picks a destination folder
- **Delete** — with confirmation

**3b. Customizable top-level structure**

In Settings, let the user define their top-level directory names:
```
Default structure:
├── Projects/
├── People/
├── Meetings/
├── Research/
└── Archive/

Custom structure (user-defined):
├── Work/
├── Personal/
├── Learning/
└── Archive/
```

This maps to Parachute note path prefixes. The sidebar reads from Settings to determine labels and ordering.

**3c. Note creation with path picker**

When creating a new note via `+ New`, show a path picker:
- Start typing → autocomplete from existing directories
- Or create a new path inline
- Default to the currently selected directory in the sidebar

**3d. Drag-and-drop reorg**

Drag a note from one folder to another → updates the note's `path` in Parachute. This is a PATCH to the API changing the path field.

### Key Files
- `src/components/navigation/ProjectTree.tsx` — add context menu, drag-drop
- `src/components/navigation/NewContentMenu.tsx` — add path picker
- `src/app/stores/settings.ts` — customizable directory labels
- `src/components/layout/Settings.tsx` — directory config section

## 4. Email via Matrix

### Problem
Currently email uses the `gog` CLI directly, which is separate from the Matrix messaging stack. Using Matrix for email (via a bridge) would unify all communications.

### Considerations
- **mautrix-email** or **email2matrix** bridges exist but are less mature than WhatsApp/Telegram bridges
- Gmail API via `gog` is reliable and already working
- Calendar and Docs still need Google OAuth regardless

### Recommendation
**Keep email via `gog` CLI for v0.1.1.** Unifying under Matrix can be a v0.2 goal when email bridges mature. The inbox redesign (section 2) should still show email alongside Matrix messages.

## 7. Architecture Decisions

### Streaming: Claude CLI vs API
- **Claude CLI streaming** (`claude -p --output-format stream-json`): Works with existing auth, has MCP access, but slower startup
- **Anthropic API streaming**: Faster, direct token streaming, but needs API key management and loses MCP tools
- **Recommendation**: Use Claude CLI streaming for v0.1.1 (consistency with existing setup). Consider API migration for v0.2 if latency is a problem.

### Message storage: Matrix vs Parachute
- **Matrix PostgreSQL**: Authoritative for full message history, real-time sync, E2E encryption
- **Parachute SQLite**: Better for search, knowledge graph integration, AI processing
- **Recommendation**: Matrix is the source of truth. Parachute indexes important/tagged messages for graph integration. Agent processes Matrix events and creates Parachute notes for actionable items.

### Contact linking: Automated vs Manual
- **Automated**: Agent matches Matrix users to Parachute person notes by name similarity
- **Manual**: User explicitly links contacts in the UI
- **Recommendation**: Start with automated matching (fuzzy name match), allow manual corrections. Store mappings in person note metadata.

## 8. Metadata Rendering Overhaul

### Problem
The Meta tab shows raw JSON for note metadata. Tag schema fields (status, priority, project, due, attendees, etc.) should render as beautiful, editable form controls — not a JSON blob.

### Design

**Schema-aware metadata rendering:**

When a note has tags with known schemas, render each field according to its type:

| Field Type | Rendered As |
|-----------|------------|
| `string` with enum values | Dropdown selector (e.g., status: todo/in-progress/done) |
| `string` (free text) | Inline text input |
| `date` | Date picker |
| `array` | Tag chip list with add/remove |
| `number` | Number input |
| `boolean` | Toggle switch |

**Layout:**
```
┌─────────────────────────┐
│ META                    │
│                         │
│ Type: Document ▼        │
│ Path: vault/projects/.. │
│                         │
│ ── task ──────────────  │
│ Status:  [In Progress ▼]│
│ Priority: [High ▼]      │
│ Project: [OpenCivics]   │
│ Due:     [2026-04-15 📅]│
│ Assigned: [Benjamin ×]  │
│                         │
│ Tags: task, opencivics  │
│ Created: Apr 10, 2026   │
│ Updated: Apr 11, 2026   │
│ Words: 1,202 · 7 min    │
│                         │
│ Sync: Google Docs ✓     │
└─────────────────────────┘
```

**Implementation:**
1. Fetch tag schema via Parachute MCP `describe-tag` or a new Tauri command `vault_get_tag_schema`
2. For each tag on the note, render its schema fields as form controls
3. Changes save back to note metadata via `vault_update_note`
4. Raw JSON view available as a toggle for power users

### Key Files
- `src/components/layout/MetadataPanel.tsx` — major rewrite
- `src-tauri/src/commands/vault.rs` — add `vault_get_tag_schema` command
- `src/app/hooks/useParachute.ts` — add `useTagSchema(tag)` hook

## 9. Fix Broken Renderers & Features

### 9a. Website Renderer — Currently broken
The website renderer crashes or doesn't render. Fix:
- Use an iframe with `srcdoc` for HTML preview (more reliable than `contentDocument.write`)
- Support multi-file projects (HTML + CSS + JS)
- Add a "Preview" button that opens in external browser
- Consider embedding a lightweight web server for hot reload

### 9b. Task Creation — Broken metadata
Creating a task via `+ New → Task` doesn't create proper metadata aligned with the `task` tag schema. Fix:
- Task creation dialog should show schema-aware fields: status dropdown, priority dropdown, project picker, due date picker, assigned text input
- Pre-populate with defaults from the tag schema
- Auto-add the `task` tag
- The current minimal text input is insufficient

### 9c. Email Sending — `gog` not found
Error: `Failed to run gog: No such file or directory (os error 2)`. The Rust `GoogleClient` resolves `gog` path at startup using `which gog`, but in production builds the PATH is different. Fix:
- Check multiple paths: `/opt/homebrew/bin/gog`, `/usr/local/bin/gog`, `~/.cargo/bin/gog`
- Store the resolved path in `AppConfig`
- Show helpful error in UI if `gog` isn't installed (with install instructions)

### 9d. Code Renderer — Upgrade to embedded IDE
The current code renderer is a plain textarea with line numbers. Replace with:
- **Option A:** Monaco Editor with WebView worker fix (investigate WKWebView web worker support in Tauri 2.x)
- **Option B:** Fork/embed CodeMirror 6 (lighter than Monaco, better WebView compatibility)
- **Option C:** Integrate with an OSS editor like Zed's editor component
- Add a code execution panel (run Python, Node, shell scripts)
- Syntax highlighting, error gutters, minimap

## 10. Canvas Renderer — Excalidraw Fork

### Vision
A freeform spatial canvas where you can:
- Place any Parachute note as a card/node on the canvas
- Draw connections between nodes that become Parachute links
- Add sticky notes, shapes, freeform drawing
- Combine Miro-level visual collaboration with Obsidian-level data connectivity

### Why Excalidraw
- **MIT licensed** — fully forkable
- **React-based** — integrates naturally with Prism's stack
- **Proven at scale** — used by millions, battle-tested
- **Rich feature set** — shapes, text, arrows, freeform draw, collaboration-ready
- **Extensible** — custom elements API for Parachute note cards

### Architecture

**Fork approach:**
```
excalidraw (npm package)
    ↓
Prism CanvasRenderer
    ↓
Custom elements:
  - NoteCard (renders Parachute note as a card)
  - LinkArrow (creates Parachute links between notes)
  - TagLabel (shows tags as colored badges)
```

**Data model:**
The canvas state is stored as the note's content (JSON), with metadata tracking linked notes:

```typescript
interface CanvasMeta extends PrismMetadata {
  type: "canvas";
  linkedNotes: string[];  // Note IDs placed on canvas
}

// Canvas content = Excalidraw scene JSON + Prism extensions
interface CanvasScene {
  excalidraw: ExcalidrawElement[];  // Standard Excalidraw data
  prismElements: PrismCanvasElement[];  // Our custom elements
}

interface PrismCanvasElement {
  id: string;
  type: "note-card" | "link-arrow";
  noteId?: string;           // For note-card: which Parachute note
  sourceNoteId?: string;     // For link-arrow: source note
  targetNoteId?: string;     // For link-arrow: target note
  position: { x: number; y: number };
  size?: { width: number; height: number };
}
```

**When a user draws an arrow between two note cards:**
1. Excalidraw fires an element change event
2. Prism detects an arrow connecting two note-card elements
3. Creates a Parachute link: `POST /api/links { sourceId, targetId, relationship: "related" }`
4. The link is now visible in the Links tab, traversable by the agent, and part of the knowledge graph

**When a user places a note on the canvas:**
1. Open the note picker (same autocomplete as @mentions)
2. A NoteCard element appears on the canvas showing the note's title and preview
3. Double-click to open the note in a tab
4. The canvas metadata tracks which notes are on it

### Implementation Plan

1. **Install Excalidraw** — `npm install @excalidraw/excalidraw`
2. **Create CanvasRenderer** — wraps `<Excalidraw>` component with Prism styling
3. **Custom NoteCard element** — render Parachute notes as draggable cards
4. **Arrow → Link creation** — detect arrow connections, create Parachute links
5. **Note picker** — `@` trigger or button to search and place notes
6. **Persistence** — save scene JSON as note content, linked note IDs in metadata
7. **Theme integration** — dark/light mode for Excalidraw canvas

### Dependencies
- `@excalidraw/excalidraw` — ~2MB (significant but worth it for the functionality)
- May need `react-dom` version alignment

### Key Files
- `src/components/renderers/CanvasRenderer.tsx` — new renderer wrapping Excalidraw
- `src/components/canvas/NoteCard.tsx` — custom element for note cards
- `src/components/canvas/NotePicker.tsx` — autocomplete to place notes on canvas
- `src/components/renderers/Registry.ts` — register for `canvas` content type
